@openclaw/relay — Install & Onboarding Mockup

Terminal flow reference · matches real openclaw configure aesthetic

Scenario 1
Install the plugin
terminal
~ openclaw plugins install @openclaw/relay ┌ Installing plugin @openclaw/relay │ │ ✓ Resolved @openclaw/relay@1.0.0 │ ✓ Installed ws@8.18.0 │ ✓ Registered channel: relay │ └ ✓ Plugin installed Run openclaw channels setup relay to connect your agent.
The plugin is loaded via jiti — no build step. Registers the relay channel automatically.
Scenario 2
First-time agent setup — one prompt, that's it
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Agent Setup │ ◆ Relay agent token │ rla_xxxxxxxxxxxxxxxxxxxxxxxx │ └
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Agent Setup │ ◇ Relay agent token │ rla_live_a8f2k9m3p7x1q5n6... │ ◆ Testing connection... │ Opening WebSocket to relay.ckgworks.com │ │ └
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Agent Setup │ ◇ Relay agent token │ rla_live_a8f2k9m3p7x1q5n6... │ ◇ Connection test │ ✓ Authenticated as Athena │ wss://api.relay.ckgworks.com/v1/ws/agent │ └ ✓ Setup complete Run openclaw gateway restart to activate Relay.
Server URL is hardcoded in the plugin — wss://api.relay.ckgworks.com/v1/ws/agent. Token input is visible as typed. No hidden characters, no env var hints. Connection test opens a real WebSocket to verify the token before saving.

~ openclaw configure ◇ What would you like to configure? │ Channels │ ◇ Select a channel │ Relay (@ckgworks) │ ◆ Relay (@ckgworks) — Athena │ │ Token rla_live_a8f2k9m3•••••••••••• │ Server wss://api.relay.ckgworks.com │ Status connected │ │ ● Test connection │ ○ Rotate token │ ○ Disconnect │ ○ Back │ └
terminal
~ openclaw configure ◇ Select a channel │ Relay (@ckgworks) │ ◇ Relay (@ckgworks) — Athena │ Test connection │ ◆ Connection test │ ✓ WebSocket open │ ✓ Authenticated as Athena │ ✓ Relay is reachable │ └ ✓ All checks passed
Relay appears in the channel list as Relay (@ckgworks) — distinct branding, separate from Flow. The detail panel shows the token with partial masking (prefix visible), server URL, and current status.
Scenario 4
Token rotation — re-run setup on an already-configured agent
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Already configured │ Agent Athena │ Token rla_live_a8f2k9m3•••••••••••• │ ◆ What would you like to do? │ ● Rotate token │ ○ Test connection │ ○ Cancel │ └
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Token Rotation │ ◇ Action │ Rotate token │ ◆ New Relay agent token │ rla_xxxxxxxxxxxxxxxxxxxxxxxx │ └
terminal
~ openclaw channels setup relay ┌ Relay (@ckgworks) — Token Rotation │ ◇ New Relay agent token │ rla_live_new_k9p2m3x7q5n8r... │ ◇ Connection test │ ✓ Authenticated as Athena (token refreshed) │ └ ✓ Token updated Run openclaw gateway restart to activate the new token.
Re-running openclaw channels setup relay on an already-configured instance detects the existing token, shows a choice, and only prompts for the new token if Rotate is selected. The connection test verifies the new token before writing config.
Scenario 5
Gateway restart after setup
terminal
~ openclaw gateway restart ┌ Restarting OpenClaw Gateway │ │ ✓ Stopped previous gateway process │ ✓ Loading config from ~/.openclaw/openclaw.json │ ✓ Channel relay → WebSocket open Athena connected │ ✓ Channel flow → webhook registered │ └ ✓ Gateway running — 2 channels active
Relay channel opens a persistent WebSocket at startup. The gateway log confirms which agent connected.
Reference
Config written to ~/.openclaw/openclaw.json after setup
~/.openclaw/openclaw.json
{ "channels": { "relay": { "token": "rla_live_a8f2k9m3p7x1q5n6..." } } }
Only the token is stored — the server URL is hardcoded in the plugin. No agent_id, no hmac_secret, no url field. Clean and minimal.
